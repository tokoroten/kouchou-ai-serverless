## 総評

一次資料の核 — クラスタを固定分類ではなく「重み付けから都度生成されるビュー」とし、
スライダー操作では**候補辺の再重み付けだけ**を行う — は技術的に正しい。
2次元座標でクラスタを決めない、UMAP のウォームスタート、ヒステリシスによる
クラスタID安定化も妥当。規模目標(数千〜1万点)も現実的
(5,000点 × 候補64辺 = 32万辺の再重み付けはフレーム内、Louvain は数十ms)。

勝敗を分けるのは以下の3点。フロントの見た目(metaball 等)は最後でよい。

## 必須の修正・明確化(一次資料への差分)

### 1. Phase 0: stance 構造化抽出の品質検証を最初に行う

このシステムの成立条件は「stance 9分類・二重否定・意見主体・confidence を安定して
抽出できるか」。ここが崩れると下流すべてが壊れた入力の上で動く。

- 一次資料末尾の代表9文 + 実データ100件程度に**人手の正解ラベル**を付けた評価セットを作る。
- モデル × プロンプトで一致率と confidence の較正(高confidence群の正解率)を測る。
- 単独指標: 「非反対」の識別、引用(holder≠筆者)の再現率。
- 合格ライン例: 隣接誤り込み一致 85%、引用検出の再現率 90%。
  届かない場合は stance を5分類に粗くする(スキーマはその縮退を最初から許す設計に)。
- Phase 0 の実出力を fixtures に流用する(フロントを偽の前提で最適化しないため)。

Phase 0 はフロントのプロトタイピング(一次資料の実装順序 1〜11、fixtures 駆動)と並行できる。

### 2. 候補 kNN グラフは「ブロック別 kNN の和集合」として定義する

一次資料の唯一の構造的な穴。semantic 距離だけで候補50〜100近傍を取ると、
stance 重みを最大にしたとき新しい距離空間での真の近傍が候補集合になく、
スライダーを動かしても分裂しない(または偽の構造が出る)。

候補集合 = semantic kNN(k=50)∪ 同一 topic 点からのサンプル
∪ stance 分布近傍(7次元なので全点計算しても安い)∪ reason タグ一致。
どの重み設定でも真の近傍がほぼ候補内にあることを担保する。
**この定義は graph-builder 実装前に確定すること**(後から変えると全部に波及する)。

### 3. stance 重みのグローバル適用は提供しない(focus+context は必須要件)

「原発賛成」と「再エネ賛成」は stance 分布が同一であり、全体空間での stance
重み付けは無関係なトピックを賛否で混ぜる無意味な配置を生む。
一次資料の「選択クラスタだけ局所再配置」は UX の好みではなく**意味論上の必須制約**。
stance 類似度はトピック条件付きでのみ有効化する。

## ブラウザ実装への置換表

| 一次資料の記述 | ブラウザでの現実解 |
|---|---|
| Leiden | 良い JS 実装がない。**graphology + Louvain** で開始(重み付き対応、数千点で数十ms) |
| HDBSCAN(初期トピック) | kNN グラフ + Louvain で代替。命名は c-TF-IDF 相当(タグ頻度の対数オッズ) |
| Wasserstein / JS divergence | 7クラスは順序付きなので**累積分布の L1**(=1次元 Wasserstein)を既定に。最速 |
| UMAP への方向制約(損失注入) | umap-js は損失をカスタムできない。**各 step 後に `x += λ(stanceScore·s − x)` と座標をナッジ**して同等の効果を得る |
| topic/reason 用の別 embedding | フェーズ2 MVP では持たない(コスト・容量3倍)。**コードブックに対する重み付きタグベクトル(疎)**で開始 |
| バックエンドで候補kNN生成 | 〜1万件なら全処理ブラウザ内で成立。Worker で実行 |

topic/reason タグは**2パス方式**を仕様とする:
1パス目で自由生成 → 統合してコードブック確定 → 2パス目でコードブックに対して割当。
(一次資料の「初期コードブック生成」の明確化。野放しだと語彙が発散しタグベクトルが比較不能になる。)

## 通常版との合流点(通常版側で先に仕込むこと)

1. **LLM 抽出の一本化**: `claimText` は通常版の `argument` と同一物。抽出を将来
   1回の構造化呼び出し(claim + target + topics + stance + reasons + ...)に拡張すれば、
   同じ抽出結果から「本家互換の静的レポート」と「インタラクティブ分析」の両方を作れる
   (抽出コストの二重払い回避)。通常版の extraction 実装では、
   **出力スキーマを拡張可能な形にしておく**(抽出結果の型を1箇所に集約し、
   追加フィールドが増えても下流が壊れない構造)。

   **【実装済み・2026-07 の設計判断】** フェーズ2は上記を「通常版 extraction への依存」ではなく
   **専用の投入口**として実装した(`src/phase2/extractEnrich.ts` の `extractAndEnrich`)。
   理由: 従来は「通常版 extraction(コメント→意見)」の後段で意見1件ごとに enrichment を
   別コールしていたため、2000コメント→約7500意見のとき extraction 2000コール + enrichment
   7500コールが走っていた。生コメントから **コメント1件=チャット1回** で
   「意見抽出 + stance/topics/reasons 等」をまとめて返す結合プロンプト/スキーマ
   (`buildExtractEnrichPrompt` / `EXTRACT_ENRICH_SCHEMA`)にしたことで、分類コールが消え約2000コールになる。
   - 埋め込みも通常版とは独立に**意見文へ張り直す**(通常版 extraction とテキストが変わりうるため再利用不可)。
   - チェックポイントは通常版(`projectId`)と衝突しないよう `"{projectId}-phase2"` に隔離する
     (`phase2-extract` はコメント単位)。よってフェーズ2は通常版の実行を前提にしない(単独で回せる)。
   - モデルが従来形式(`extractedOpinionList` / 文字列配列)に戻った場合も `parseExtractEnrich` が拾い、
     enrichment は unknown 立場のフォールバックにする。本家互換の静的レポート系統(通常版)は不変。
2. **ビューのスナップショット → Result 互換エクスポート**: 現在のクラスタ+
   テンプレートラベルを本家互換 `Result` JSON に変換すれば、通常版 §7.1 の
   単一 HTML エクスポートがそのまま「動かない共有用レポート」として使い回せる。
3. **永続化**: フェーズ2で保存するのは `OpinionRecord[]` + `ClusterView`(一次資料の通り)。
   通常版の IndexedDB / persist() / エクスポート基盤にそのまま載る。
4. **clustering worker のプロトコル**: 通常版 §6.3 で「中間座標(Float32Array,
   transferable)を流せるプロトコル」を既に指定済み。これがフェーズ2の
   リアルタイム再配置の土台になる。

## 留意事項(一次資料の注意事項に追加)

- クラスタリング(グラフ+Louvain)と表示(UMAP)の分離は通常版のパイプラインと
  思想が異なる(通常版は 2D 座標上で KMeans)。フェーズ2は本家互換を目的としないので
  これでよいが、**同じ画面に両モードの結果を混ぜて表示しない**こと(ユーザーが
  「同じクラスタリング」と誤解する)。モードは明示的に切り替える。
- LLM 解説(オンデマンド)はクラスタ構成(所属 id 集合のハッシュ)をキーにキャッシュする。
